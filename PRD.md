# AI Teams — 多 Agent 终端协作工作台

> 产品需求文档 v0.1 · 2026-06-11
> 一句话定义：**像群聊一样指挥多个 CLI Agent（Claude Code / Codex / Kimi …），以本地文档 + Git worktree 作为统一上下文，在一个界面里完成分发、监控与评审。**

---

## 1. 背景与痛点

我（用户）日常的工作流是：写好一份技术文档（PRD / 设计稿 / 评审稿），然后把它交给多个不同的 CLI Agent 并行处理——例如让 Codex 实现、Claude Code 评审、Kimi 提意见。当前体验的核心痛点：

| # | 痛点 | 具体表现 |
|---|------|----------|
| P1 | **频繁切换窗口** | 每个 agent 一个 terminal 标签页/窗口，来回 cmd+tab，无法一眼总览谁在干活、谁卡住了、谁在等输入 |
| P2 | **上下文重复交代** | 每开一个 agent，都要重新口述"技术文档在哪、开发分支是哪个、约束是什么"，纯重复劳动且容易漏 |
| P3 | **没有统一的信息载体** | 各 agent 的产出散落在各自的会话里，无法沉淀为可对比、可归档的评审记录 |
| P4 | **并行工作互相踩踏** | 多个 agent 改同一个仓库会冲突，需要手动管理分支/worktree |

**核心洞察**：这些痛点的共性是缺一个"调度层"。Agent 本身已经够强，缺的是一个让人类以**广播/点名**方式分发任务、以**文档和 Git**作为共享事实源（single source of truth）的指挥界面。

---

## 2. 产品理念

三条设计原则，所有功能决策都回到这里：

1. **Chat as Router（群聊即路由）**
   一个输入框统一收口。`@claude` `@kimi` 点名发送，`@all` 广播。人不去找 agent，消息去找 agent。

2. **Docs + Git as Context（文档与 Git 即上下文）**
   不靠口头转述。每个任务对应一个**任务文档**（markdown），包含需求、分支名、worktree 路径、约束。Agent 接到的第一条消息永远是"读这个文档"。Agent 的产出也回写为文档（评审意见、实现说明），形成闭环。

3. **Terminals as First-class Citizens（终端即一等公民）**
   不重新发明 agent 的交互，不做 API 封装层。每个 agent 就是一个真实的 PTY 终端进程（跑 `claude`、`codex`、`kimi` CLI），产品只做**编排、注入、观察**。这样任何新出的 CLI agent 都能零适配接入。

---

## 3. 目标用户与场景

**目标用户**：重度使用多个 CLI coding agent 的开发者 / 技术 Lead。

### 核心场景

**场景 A：多 Agent 文档评审（最高频）**
1. 我写好 `design.md`，在输入框输入：`@all 请评审 docs/design.md，重点看并发模型，意见写到 reviews/<你的名字>.md`
2. 三个终端同时收到消息开始工作，我在一个屏幕上纵向排列实时观察
3. 各 agent 把评审写到各自的文档，我（或再指定一个 agent）汇总对比

**场景 B：并行开发 + 交叉评审**
1. 为任务创建 worktree：产品自动 `git worktree add ../proj-feat-x feat-x`
2. `@codex 在 worktree A 实现 task.md 里的功能`，`@claude 在 worktree B 实现同一个`
3. 完成后 `@kimi 对比两个分支的 diff，出一份评审报告`

**场景 C：长任务看护**
- Agent 跑长任务时我去干别的，界面侧边栏显示每个 agent 状态（工作中 / 等待输入 / 空闲 / 报错），等待输入时通知我回来处理权限确认。

## 4. 功能需求

### 4.1 MVP（先把闭环跑通）

**F1. Agent 管理**
- 添加 agent：起名（如 `claude`、`kimi`）、指定启动命令（如 `claude --dangerously-skip-permissions`）、工作目录
- 开关：每个 agent 可单独启动/停止/重启；关闭的 agent 在群聊中置灰
- 预设模板：内置 Claude Code / Codex / Kimi 等常见 CLI 的启动配置

**F2. 统一输入框 + @ 路由**
- `@name 消息` → 注入到对应终端的 stdin（模拟键入 + 回车）
- `@all` / 多个 `@` → 依次注入到多个终端
- 不带 @ → 默认发给当前聚焦的终端
- @ 自动补全（输入 @ 弹出在线 agent 列表）

**F3. 多终端布局**
- 同一窗口内 N 个真实终端（PTY + xterm 渲染），支持纵向排列 / 网格布局
- 每个终端有名字标签 + 状态指示灯
- 单击某个终端可直接接管键盘（处理交互式确认）

**F4. 任务文档（Handoff Doc）**
- 一键"新建任务"：生成模板化 markdown（目标、上下文、分支、worktree 路径、产出要求）
- 发送任务 = 自动拼装一条消息：`请先读 <doc 路径>，按文档要求工作，产出写到 <output 路径>`
- 任务文档存在项目内固定目录（如 `.aiteam/tasks/`），天然被 git 跟踪

**F5. Agent 状态检测**
- 通过终端输出的活跃度 + 模式匹配（如检测到权限提示、空闲提示符）判断状态：🟢 工作中 / 🟡 等待输入 / ⚪ 空闲 / 🔴 异常
- 等待输入时系统通知

### 4.2 V2（增强）

**F6. Worktree 编排**
- 界面内列出仓库的所有 worktree；新建任务时一键 `git worktree add`，自动写进任务文档
- 任务完成后辅助清理（merge 提示 + `worktree remove`）

**F7. 会话记录文档化**
- 每个 agent 的对话流自动落盘为 markdown（`.aiteam/sessions/<agent>/<date>.md`），可检索、可 diff、可归档

**F8. 编排模板（Playbook）**
- 把"评审流"存为模板：一键触发"@all 评审 → 收集产出 → @claude 汇总"的多步流程

**F9. 产出聚合视图**
- 自动监听 `reviews/` 等输出目录，多 agent 的产出并排对比展示

### 4.3 非目标（明确不做）

- ❌ 不做 agent 的 API 封装 / 自研 agent —— 只编排现成 CLI
- ❌ 不做云端协作 / 多人共享 —— 纯本地单人工具
- ❌ MVP 不做 agent 之间的自动对话（agent 互相 @）—— 人始终在环路中心
- ❌ 不解析/翻译 agent 的输出语义 —— 终端原样展示，文档才是结构化产出

## 5. 技术方案思路

### 5.1 架构概览

```
┌─────────────────────────────────────────────┐
│                 UI 层 (单窗口)                │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐        │
│  │term:claude│ │term:codex│ │term:kimi │  ← xterm.js 渲染
│  └─────────┘ └─────────┘ └─────────┘        │
│  [统一输入框: @claude @kimi 评审 design.md ]  │
├─────────────────────────────────────────────┤
│               编排层 (核心)                   │
│  · PTY 管理器（spawn/kill，每 agent 一个 pty） │
│  · 消息路由器（@解析 → 写入对应 pty stdin）     │
│  · 状态机（输出流分析 → 工作中/等待/空闲）       │
│  · 任务文档生成器（模板 → .aiteam/tasks/）     │
│  · Worktree 管理（git worktree 封装）         │
├─────────────────────────────────────────────┤
│              事实源（不依赖产品也能读）          │
│  · 项目仓库 + git worktrees                  │
│  · .aiteam/ 下的任务文档、产出文档、会话记录     │
└─────────────────────────────────────────────┘
```

关键点：**编排层只做三件事——把进程拉起来、把文本送进去、把输出看明白。** 所有持久信息都在文件系统和 git 里，产品挂了也不丢上下文。

### 5.2 技术选型（两条路线）

| 路线 | 组成 | 优点 | 代价 |
|------|------|------|------|
| **A. tmux 方案（最快验证）** | tmux 做 PTY 宿主，`send-keys` 注入消息，`capture-pane` 读输出；薄 UI 或纯 CLI 控制 | 一两天能跑通；tmux 本身解决了会话存活、分屏 | UI 体验受限，状态检测较粗糙 |
| **B. 桌面 App（产品形态）** | Electron/Tauri + node-pty + xterm.js | 完整 UI 控制：状态灯、@ 补全、产出对比视图 | 开发量大数倍 |

**建议路径**：先用路线 A 做一周内的可用原型验证工作流（@ 路由 + 任务文档这两个假设最值得先验证），跑顺了再投入路线 B。Tauri（Rust + webview）比 Electron 更轻，配合 `portable-pty`，适合最终形态。

### 5.3 可靠性设计（核心难点与对策）

> 设计决策：**不走 ACP / API 封装路线**，坚持纯 PTY。代价是注入和状态检测天然不可靠（现代 CLI 都是全屏 TUI，raw mode 下注入的字节被当按键解释，时机不对会被菜单/渲染吞掉；空闲时 spinner 仍持续刷屏，"静默"信号不存在）。以下三条对策把不可靠信道工程化为可验证信道：

1. **闭环注入（verify-before-enter）**：注入消息文本后先不发回车，capture 终端画面确认消息完整出现在输入行，再发回车；未出现则退避重试，重试耗尽则显式告警（绝不静默失败）。实现上用 tmux `load-buffer` + `paste-buffer -p`（bracketed paste）而非逐键 send-keys。消息体保持单行短句，长内容一律走文档引用——这也反向强化"文档为事实源"。
2. **钩子上报状态，启发式兜底**：优先用各 CLI 自带的进程外通知机制写状态文件（`.aiteam/status/<agent>.json`）——Claude Code 用 hooks（Stop/Notification 事件），Codex 用 `notify` 配置；编排层 watch 文件即可获得精确状态，且与"文件系统为事实源"理念一致。无钩子机制的 agent 降级到"输出区域 diff + 末行模式匹配"启发式，状态灯标注为"推测"。注意：模式匹配规则按 agent 版本维护，CLI 更新后文案变化是常态，需要可热更新的规则配置。
3. **交互式确认**：agent 弹权限确认时需要人接管。对策：F3 的"点击终端直接接管键盘" + F5 的通知（钩子上报的"等待输入"状态触发），足够 MVP。

## 6. 竞品 / 参考

- **Cumora**（cumora.ai，"where agent teams gather"）：群聊式人机协作工作台，agent 有角色/记忆/状态、可互相 DM——验证了"群聊即指挥界面"的交互范式，是本产品 UI 心智的直接参照。但它的 agent 是自建的对话角色，**不是本地终端进程**，不接 Claude Code / Codex 等 CLI，也不以文档和 Git 为事实源。本产品 = Cumora 的群聊交互 × 真实 PTY 终端 × 文档/Git 上下文。
- **Conductor / Crystal**（Mac 上的并行 Claude Code 管理器）：验证了"worktree + 多实例并行"需求真实存在，但都绑定单一 agent；本产品的差异点是**跨厂商 agent 群聊式编排**。
- **claude-squad / vibe-kanban**：开源的多 agent 终端管理，偏任务看板形态；本产品差异点是**统一输入框的 @ 路由**与**文档作为 handoff 协议**。
- **tmux + 手工脚本**：现状的天花板，证明路线 A 技术可行。

差异化定位一句话：**别人管的是"多个并行的终端"，我们管的是"一个团队"——有名字、能点名、用文档交接。**

## 8. 待决问题（Open Questions）

1. **MVP 形态**：先做 tmux 原型验证，还是直接上桌面 App？（建议前者——闭环注入和钩子状态上报这两个可靠性机制，在 tmux 原型阶段就能完整验证）
2. **桌面框架**：Tauri vs Electron？（倾向 Tauri，但若想快速复用 xterm.js 生态，Electron 资料更多）
3. **任务文档的 schema**：固定 frontmatter 字段（branch / worktree / output_path）还是自由 markdown？（建议固定 frontmatter + 自由正文；另需区分"指令"与"待评审材料"两类内容，降低评审链中 agent 产出对下游 agent 的间接 prompt injection 风险）
4. **要不要支持 agent 互相可见**：例如 claude 的评审让 kimi 反驳——这会显著放大价值但也放大失控风险，建议 V2 以后再考虑。
5. **"已读回执"机制**：handoff 依赖 agent 真的读了文档且是最新版，但无强制力。候选缓解：任务文档要求 agent 在产出文件开头先复述任务目标，作为已读证明。
