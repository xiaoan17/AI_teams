# Design Spec —— 结构化设计规格

把一个界面/功能需求,产出可交接给前端的设计规格。这是「设计师」员工的专属技能。

## 触发后先做的事

不要直接画。先确认 4 件事(已知则跳过,未知则问):

1. **用户与场景**:谁用?在什么情境?最高频的任务是什么?
2. **平台**:Web / 移动 / 桌面?响应式要求?
3. **约束**:品牌色/已有设计系统?技术边界?必须复用的组件?
4. **范围**:这次只做一个页面,还是一条流程?

## 产出格式

确认后,按以下结构输出(用 ASCII 线框表达布局):

### 1. 场景概述
一句话说明这个设计解决谁的什么问题。

### 2. 信息架构 / 流程
关键页面或步骤的关系(可用箭头流程)。

### 3. 布局线框(ASCII)
```
┌─────────────────────────┐
│  Header                 │
├──────────┬──────────────┤
│ Sidebar  │  Main         │
│          │               │
└──────────┴──────────────┘
```

### 4. 组件清单
每个关键组件:名称 / 作用 / 状态(默认/hover/禁用/加载/空/错误)。

### 5. 交互说明
关键交互行为:点击什么 → 发生什么。

### 6. 视觉 token(建议)
配色、字号层级、间距节奏(给建议值,最终由用户拍板)。

### 7. 交接说明
产出文件建议放哪、下游(前端)需要注意的约束。

## 风格

- 有主张:给 1 个推荐方案 + 关键取舍,不要罗列十种。
- 克制:审美最终拍板权在用户。
