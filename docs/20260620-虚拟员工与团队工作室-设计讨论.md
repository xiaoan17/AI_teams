# 从「多 Agent 面板」到「可雇佣的虚拟员工团队」—— 设计讨论稿

> 这是一份**讨论稿**，记录概念模型、架构方案、权衡与决策。**核心方向已于 2026-06-20 拍板（见 §0.5）**，M1 已按该方向落地。

---

## 0.5 已拍板的架构定论（2026-06-20）

经 M0 实测 + 讨论，核心架构已确定，后续实现以此为准：

> **全局库 `~/.aiteam/roles/<员工>/` 是模板；雇佣时实例化拷贝到项目 `.aiteam/crew/<员工>/`（物理独立、可 git、项目间隔离）；启动时工作目录 = App 加载的项目根（不污染项目），用 `--add-dir <crew副本>` 注入专属 skill、`--append-system-prompt <人设>` 注入人设。协作先人工（`@员工` + tasks handoff），不做自动调度。**

| 决策点 | 定论 | 依据 |
|--------|------|------|
| 目录方案 | **B 实例化拷贝**（全局模板 → 项目 crew 实例） | 比软链稳，跨机器/打包/git 不碎；项目间隔离 |
| 工作目录 vs 人设目录 | **分离**：工作目录=项目根，人设/skill 从 crew 目录**注入**（非 cp 进工作区） | M0 实测 `--add-dir`+`--append-system-prompt` 可行且零污染 |
| 注入方式 | `--add-dir <crew>`（skill）+ `--append-system-prompt`（人设） | 多员工同项目时天然隔离，避免 `.claude/` 互相覆盖 |
| 协作模型 | **先人工**（`@员工` / `@all` + `.aiteam/tasks/`），自动编排留后期 | 呼应双轨：spec/验证本就该吃人工 |

> 为什么不直接 cp 人设进工作目录（你最初的想法）：能跑，但①会往项目根塞 `.claude/`+`CLAUDE.md`，可能和项目自己的冲突；②一个项目同时雇多个员工时，几份 `.claude/` 会互相覆盖打架。注入方式两者皆免。

---

## 0. 一句话目标

把 AI_teams 从「**几个固定的 agent CLI 并排跑**」升级成「**一个能养一整支虚拟员工队伍的工作室**」：

- 有一个**全局员工库**（总经理、PRD、画图、前端、测试……每个员工是一套人设 + skill 组合）。
- 每个项目可以**雇佣**其中几个员工，雇佣关系是项目级的。
- 每次启动时，App 弹一个**选择器**，让我勾选「这次开工请谁」。
- 被请来的员工，加载的是**它专属的、项目级的** persona 和 skill —— 只有这个员工自己能体验到。

## 1. 已确认的四个方向（本稿的前提）

| 决策点 | 选择 | 含义 |
|--------|------|------|
| 员工底层 | **同一个 CLI（主要 Claude Code）换不同人设** | 不为每个员工绑死一个独立 CLI |
| 员工归属 | **全局库 + 项目实例化** | 全局可复用、可分享；项目里「雇佣」并可本地覆盖 |
| 启动交互 | **App 内可视化选择器** | 启动时弹「员工花名册」勾选 |
| skill 含义 | **复用 CLI 原生 skill** | 员工的 skill = Claude Code 原生 `.claude/skills/` 那套，不另造提示词体系 |

## 2. 关键技术发现（决定了整个方案能多优雅）

读了 `aiteam.py` 和现有 agent schema 后，有一个**核心杠杆**：

> Claude Code 加载哪些 skill / persona，是由它**启动时的工作目录 `cwd`** 决定的 ——
> 它会读 `cwd/.claude/skills/`、`cwd/CLAUDE.md`、`cwd/AGENTS.md`。

而我们现有的 agent schema **已经有 `cwd` 字段**（见 `.aiteam/agents.json`），`tmux new-session -c <cwd>` 也已经在用它。

M1 前的最初判断是「换员工 ≈ 换 cwd」；M0/M1 后的最终实现把工作目录与人设目录分离，技术内核调整为：

```
一个「员工」  =  一个项目 crew 副本 + role 元数据 + 一段人设提示词
「雇佣某员工」 =  全局模板复制到项目 .aiteam/crew/<员工>
「启动某员工」 =  工作目录仍是项目根，CLI 追加 --add-dir <crew> + --append-system-prompt <人设>
```

四个已确认方向因此天然自洽：
- **同 CLI 换人设** → 命令都是 `claude`，只是注入的 crew/persona 不同 ✓
- **复用原生 skill** → crew 内 `.claude/skills/` 通过 `--add-dir` 注入 ✓
- **全局库 + 项目实例化** → 全局放员工模板目录，项目里实例化（拷贝/软链）✓
- **App 选择器** → 选员工 → 决定本次启动注入哪个 crew ✓

> ✅ **地基已验证（2026-06-20，M0 实验通过）**：用真实 Claude Code 2.1.177 在实例化的 `.aiteam/crew/designer/` 目录里 headless 启动，确认：
> 1. `cwd/CLAUDE.md` 作为该工位人设被加载（Claude 自称「产品设计师」并复述人设）；
> 2. `cwd/.claude/skills/design-spec/` 项目级专属 skill 被加载且能准确说出用途；
> 3. 全局 `~/.claude/skills/`（zotero / lark 等）**仍可用，未丢失**。
>
> 结论：**项目级 skill 与全局 skill 是「叠加 merge」关系，非互斥**。整个方案地基成立，且「换员工 ≈ 换 cwd」的预测正确。

## 3. 概念模型（词汇表，先统一语言）

为避免「agent / 员工 / 角色 / skill」混着说，先定义清楚：

| 概念 | 定义 | 类比 |
|------|------|------|
| **Role（岗位 / 员工模板）** | 全局库里的一个员工定义：人设描述 + 一套 skill 组合 + 默认 CLI/参数 | 公司的「岗位说明书」 |
| **Hire（雇佣 / 员工实例）** | 某项目里被实例化的一个 Role | 「这个项目里入职的那位 PRD」 |
| **Roster（花名册）** | 某项目当前雇佣的所有 Hire 的集合 | 「这个项目的团队名单」 |
| **Workspace（项目）** | 当前打开的代码项目 | 「一家分公司 / 一个项目组」 |
| **Crew Picker（员工选择器）** | 启动时勾选「这次请谁」的 UI | 「今天叫哪几位来开工」 |

> 命名只是占位，欢迎你改。但「Role（模板）vs Hire（实例）」这条线我建议保留 —— 它正好对应你说的「全局库 + 项目实例化」。

## 4. 目录与数据布局（三种方案）

这是最需要你拍板的地方。三种方案在「复用性 ↔ 项目隔离 ↔ 实现复杂度」上取舍不同。

### 方案 A：软链（symlink）—— 全局库单一真相源

```
~/.aiteam/roles/                      # 全局员工库（单一真相源）
  general-manager/
    role.json                         # 人设、默认 CLI、描述
    .claude/
      skills/...                      # 这个员工的原生 skill
    CLAUDE.md                         # 这个员工的 persona
  prd/
  diagrammer/

<project>/.aiteam/crew/               # 项目花名册（只是指针）
  general-manager -> symlink 到全局   # 软链回全局库
  prd/                                # 或：项目本地覆盖版（实拷贝）
```

- **优点**：员工模板改一次，所有项目同步生效；项目里几乎零冗余。
- **缺点**：软链跨机器/打包/git 容易碎；「项目级独立内容」被弱化（改全局会影响别的项目）。

### 方案 B：实例化拷贝（copy-on-hire）—— 项目级真正独立 ✅ 已采用

```
~/.aiteam/roles/general-manager/...   # 全局库 = 模板（只读心智）

<project>/.aiteam/crew/general-manager/
  role.json                           # 雇佣时从全局库拷贝过来
  .claude/skills/...                   # 项目本地副本，可单独改
  CLAUDE.md
  .source                             # 记录「来自哪个全局模板 + 版本」，便于将来 diff/更新
```

- **优点**：完全契合你说的「**独立的、项目级、只面向当前 agent**」；可单独魔改不影响全局；可 git 提交跟项目走。
- **缺点**：全局模板更新后，项目副本不会自动跟进（需要一个「同步/更新」动作，但这其实是好事 —— 显式可控）。

### 方案 C：叠加层（overlay）—— 全局为底 + 项目补丁

启动时把「全局 role 目录」+「项目覆盖目录」合成一个临时 `cwd`（用 overlay 或拼装）。

- **优点**：既共享又可覆盖，冗余最小。
- **缺点**：实现最复杂（要做目录合成 / 临时工作区生命周期管理），调试困难。当前阶段**不建议**，作为远期选项记录。

> **我的倾向：方案 B**。它最贴合你原话里的「独立的项目级内容、只面向当前启动的这个 agent」，且实现直白、可 git 化、无软链脆弱性。代价（模板更新不自动）反而符合工程直觉。

## 5. Role 定义的 schema（草案）

在现有 agent schema（`id/name/command/args/cwd/enabled/permission_mode`）上**向后兼容地扩展**，新增字段都可选，老配置不受影响。**关键变化**：员工的 `cwd` 不再指向自己的目录——工作目录跟随 App 加载的项目根，人设/skill 通过启动参数注入：

```jsonc
{
  "id": "designer",
  "name": "设计师",
  "role": {                              // ← 新增：员工身份
    "title": "产品设计师 / 视觉与交互",
    "summary": "把需求转成界面设计、交互流程与示意图",
    "emoji": "🎨",
    "track": "spec"                      // 呼应双轨：spec / impl / verify（可选）
  },
  "command": "claude",
  "args": ["--dangerously-skip-permissions"],
  "skills": ["design-spec"],             // ← 声明用到的 skill（展示/校验用）
  "persona_dir": ".aiteam/crew/designer",// ← crew 实例目录；启动时 --add-dir 注入它（带 .claude/skills）
  "persona_file": "CLAUDE.md",           // ← persona_dir 内的人设文件；启动时 --append-system-prompt 注入
  // 注意：不再写 "cwd" 指向员工目录。工作目录 = App 加载的项目根。
  "enabled": true,
  "permission_mode": "configure-before-start"
}
```

启动时（伪代码）实际拼出的命令：

```bash
cd <项目根>   # 工作目录 = App 加载的项目，看得到全部代码
claude --dangerously-skip-permissions \
  --add-dir <persona_dir> \                          # 注入专属 skill
  --append-system-prompt "$(cat <persona_dir>/<persona_file>)"   # 注入人设
```

要点：
- `command` 仍是 `claude` —— 同 CLI 换人设。
- 「这个员工独立的 skill / persona」物理存在 `persona_dir`（crew 实例目录），通过**注入**生效，**不污染项目工作目录**。
- 多员工同项目时各自 `--add-dir` 自己的目录，天然隔离。
- `track` 字段是和 §双轨文章 的接口（spec/impl/verify），现在可先不用，留好钩子。

## 6.5 启动体验：「3 个员工工位」（2026-06-20 定）

把现有的「3 个 agent 面板」**语义升级**成「3 个员工工位」——底层 tmux/PTY 编排不动，动的是配置层、启动参数、标题渲染、选择器 UI。三点诉求及落地：

### 诉求 ① 整个对话框 = 那个员工的注入
- **现有架构天然支持**：每个面板背后是独立 tmux window + 独立 `claude` 进程。启动那一刻注入 `--add-dir <crew> --append-system-prompt <人设>`，该进程整个生命周期就是那个员工，不中途变身。
- 落点：`agentShellCommand()`（main.cjs / aiteam.py 对应函数）按员工拼注入参数。

### 诉求 ② 关掉要重新启动配置（不持久复活）
- 定论：**「不复用但不急杀」**。关 App 不主动 kill tmux（保留现有「关窗不杀 agent」设计，那是持久化价值，也踩过坑）；但**下次启动不读「上次选了谁」、不自动重连，始终重新弹选择器**。
- 关键认知：② 是**会话层概念，不是进程层**。「散场」= 下次不复用旧的组队选择，而非杀进程。
- 落点：组队选择结果**不作为持久状态复活**；现有僵尸 session 回收逻辑照常清理旧 session。

### 诉求 ③ 窗口名带职位名（好辨认）
- 定论：**UI 标题显职位**。
- ⚠️ 坑：tmux `window_name` 当前是 **pane 匹配键**（`tmux-runtime.cjs` 靠它反查 pane），**不能**改成中文职位名（且会撞「tmux 改写点/特殊字符」老坑）。
- 落点：tmux 窗口名**保持稳定 id 不动**；App 面板标题（renderer）渲染 `role.title`（如 `🎨 设计师`）。安全、无副作用。

### 完整启动流程（三工位各选员工）
```
打开项目 → 点「开工」
  ↓ 弹【组队选择器】(融合进现有 3-agent 配置界面)
  工位1 [▼选员工]→🎨设计师   工位2 [▼选员工]→📋PRD   工位3 [▼选员工]→🧭总经理
  ↓ 确认：对每个工位 雇佣(模板→crew) + 注入启动 claude
  3 个面板，标题 🎨设计师 / 📋PRD / 🧭总经理；每面板整段对话都是该员工(①)
  ↓ 关掉 App → 不复用本次组队，下次重新选(②)
```

> 三工位可同时是不同员工，也可都是同一个员工。这正是现有「选不同 agent 组成 3 个」的能力，只是把「agent」换成「员工」。

### M1 实现回填（2026-06-20）

- `docs/plans/20260620-虚拟员工M1-plan/role-schema.md` 成为 role/agent 扩展字段契约。
- 全局库已包含 `designer`、`manager`、`prd` 三个模板。
- CLI 支持 `role list`、`role hire <id>`、`start --role <id>`；`start --pick` 提供交互选择。
- CLI 与 App 启动命令均按 `persona_dir` 注入 `--add-dir` 与 `--append-system-prompt`，工作目录保持项目根。
- App sidebar 的 Start 入口打开三工位组队选择器；面板标题显示 `role.emoji + role.title`。
- tmux window name 保持 agent id，不使用中文职位名。
