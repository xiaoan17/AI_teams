# 你不需要十个 Agent，你需要两条轨道

> 原文：[You don't need ten agents. You need two tracks.](https://x.com/i/article/2067686501891112960) — Hugo Baraúna（Dashbit 开发者倡导者，Elixir Radar 主编，Tidewave / Livebook）

## 核心观点

网上常看到有人炫耀终端里同时跑十个 coding agent，暗示「Agent 越多，生产力越高」。作者不买账——**与其开十个 agent，不如只开两个，分成「规格轨」和「实现轨」两条并行轨道**。

这套工作流被作者用来构建真实生产软件：Elixir Radar 邮件订阅平台的后台系统（内容策展、邮件发送、客户自助赞助管理）。

---

## 两条轨道

### 1. 规格轨（The Spec Track）—— 需要你高度专注

起点是一个 feature 想法（可能只有一两句话），终点是一份完整的功能规格（spec）+ 技术设计（plan）。

**两步走：**

1. **功能规格（PRD）**
   - 让 agent 帮你头脑风暴
   - agent 会反过来问你问题，逼你澄清思路
   - agent 阅读代码库理解现状
   - 最终产出一份功能需求文档（PRD）

2. **技术设计 + 实现计划**
   - 基于功能 spec，让 agent 产出技术架构设计
   - agent 同样会读代码库、提问、等你确认
   - 最终产出包含「技术设计 + 拆解成工程任务」的实现计划

> 这一步不能偷懒：不能直接丢一句「帮我做个 XX 功能」就让 agent 去跑——输入太薄，后续实现必然跑偏。正因为它需要持续的人工参与，**spec 轨很难并行**。

### 2. 实现轨（The Implementation Track）—— 可半自主运行

拿到 spec 轨的产出后，对 agent 说：「这是 spec 和实现计划，请实现它。」

- 不需要全程盯——agent 已有足够输入自主工作一段时间
- 可以选择「每个任务完成后 review」或「最后统一 review」
- **并行机会**：agent 实现第一个 feature 时，你立刻启动下一个 spec

**循环节奏**：等你 review 完上一个实现时，新 spec 往往已写完或接近完成 → 再开一个实现 agent → 持续滚动。

## 为什么只用两个 Agent？

### ① 平衡两种「注意力需求」

| 轨道 | 人工注意力 | 能否并行 |
|------|-----------|---------|
| 规格轨 | 高（需持续问答、澄清、review） | 难 |
| 实现轨 | 低（输入齐备后可自主跑） | 易 |

两条轨互补：一个吃注意力、一个不吃，正好能同时进行。若硬开两个 spec 轨并行，会难以驾驭。

### ② Spec 是第一个瓶颈

假设你有十个实现 agent，但只有你一个人在写 spec。**实现 agent 的速度上限 = 你喂 spec 的速度**，而你一次只能写一份 spec，不是十份。

### ③ 「写完代码」不等于「完成工作」

即便你真写出十份独立 feature spec 并行喂给十个 agent，代码生成完后仍会撞上人工瓶颈：

- Code review
- 功能性人工测试
- UI 改进

软件开发有三阶段：**Specification → Implementation → Verification**。Agent 能在实现阶段自主，但**规格和验证阶段仍需大量人工**——对不是瓶颈的环节做并行化，并不能让整体变快。

### ④ UX 无法外包

面向客户的功能，要做出好产品，UX 不能交给 agent。前端代码 agent 能写，但 UI 迭代是主观的、有立场的，得来自你本人，不是来自 agent。

## 适用人群

- **Builder 型**：同时握着产品决策和代码的人——独立开发者、indie hacker、技术创始人（最大受益者）
- **普通开发者**：PM 给你一份已拍板的功能 spec，你仍需把它转成技术设计 + 实现计划；agent 实现时你可以做下一个 backlog 项的技术设计——「也许能开两条以上，但大概率开不到十条」
- **不适用场景**：注定要被丢掉的代码——spec/plan/review 的成本收不回来，**vibe coding 就够了**

## 演进，而非颠覆

这套工作流并非凭空发明，背后是成熟的软件工程与系统思维：

| 原理 | 出处 |
|------|------|
| 系统速度受限于瓶颈 | 约束理论 / 排队论 |
| 先完成再开新的并行 | Kanban 的 "Stop starting, start finishing" |
| "双轨"命名 | Marty Cagan 的 "Dual-Track Development" |

> 新工具很惊艳，但旧原则依然成立——只是把它们适配到 agentic 开发世界。

## 我的批注

- 核心洞察是**「瓶颈识别」**：与其盲目堆 agent，先想清楚哪一步吃人工注意力。实现阶段可并行，spec 和验证阶段不可。
- 「两条轨」本质是把**高耦合人工环节**与**可自主环节**解耦后并行，类似 CPU 流水线的思想。
- 对个人开发者最有参考价值的一点：**spec 轨投入越扎实，实现轨越能放手**——agent 自主性的上限由输入质量决定。
- 可对照自己当前 agent 用法自检：是不是在 spec 没写清时就让 agent 实现，导致反复返工？是不是开了多个实现 agent 却没有 spec 源源不断喂入？
