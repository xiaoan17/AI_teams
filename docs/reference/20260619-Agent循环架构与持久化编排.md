# Agent 循环架构与持久化编排

> Dan Farrelly (@djfarrelly) · 2026-06-19 · 👁 6.1万 · 💬 5 · 🔁 22 · ❤️ 128

> 核心论点：所有人都在问"循环是什么"，却没人问"究竟什么在驱动循环运转"。持久化（durability）不是循环的一个属性，而是它底下的整个执行层。

## 背景脉络

AI 圈已经形成共识：**循环（loop）是智能体系统的核心基础构件**。两位作者的近期讨论奠定了基础：

- **Matt Van Horn（@mvanhorn）**：梳理了 Agent 循环的演进谱系 —— 从 ReAct → 工具调用 → 编排循环 → 循环监督循环。最终落脚于**持久性**：无法在重启后存续的循环不算真正的循环。
- **Addy Osmani（@addyosmani）**：拆解了循环内部的组件 —— 自动化流程、工作树、技能模块、连接器、子智能体。核心论点是**编排**：构建能自主驱动 Agent 的系统，而非依赖人工持续干预。

作者将两者观点推向更深处：**持久化编排是构建 Agent 循环架构的基石**。

---

## 循环在哪里崩塌

`/loop` 和 `/goal` 这类模式能很好地处理单 Agent、单会话的场景。但在 Van Horn 框架的"第五阶段"，问题开始暴露：

- 循环监督其他循环
- 循环按计划运行（而非由人类触发）
- 循环需要在进程重启、部署、崩溃后存活
- 循环派生子 Agent 并等待结果（有时是几小时后）
- 循环需要事后可观测

**这不是提示词问题，而是基础设施问题。**

### 朴素循环为什么失败

当你在服务器上跑一个 Agent 循环时，进程总会死或重启（部署、OOM、抢占式实例回收）。循环重启后：

- 它在做什么？执行到哪一步了？
- Slack 消息已经发了吗？
- 子 Agent 已经启动了吗？

你都不知道。它从头开始：重新拉取已有数据、重新调用 LLM 做已做过的决策、发重复通知、启动重复子 Agent。**结果是三条一模一样的 Slack 消息和一个困惑的团队。**

修复方法不是"更好的错误处理"，而是一种执行模型：**每一步都有检查点，每个决策都被持久化，恢复意味着从最后一个成功步骤继续。**

## 三层 Agent 循环架构

每一层都映射到一个具体的基础原语。

### Layer 1：循环（The Loop）

**循环 = cron + 决策器**。它按计划（或触发器）运行，评估状态，决定下一步做什么。

Van Horn 定义的具体化：cron 从来没有的是中间的"决策"环节。**Agent 来决策，而不是你。** Cron 是心跳，LLM 是决策器，Steps 是持久化执行、检查进度。

```typescript
export const infraHealthCheck = inngest.createFunction(
  { id: "infra-health-check" },
  { cron: "*/30 * * * *" }, // 每 30 分钟
  async ({ step }) => {
    const metrics = await step.run("fetch-service-metrics", async () => {
      return await fetchServiceMetrics(); // 错误率、延迟、内存、CPU
    });

    const assessment = await step.run("assess-health", async () => {
      return await callLLM({
        prompt: `Given these service metrics, classify overall system health
                 as "normal", "degraded", or "critical". Explain your reasoning.
                 Metrics: ${JSON.stringify(metrics)}`,
      });
    });

    if (assessment.status === "degraded" || assessment.status === "critical") {
      await step.invoke("triage-incident", {
        function: incidentTriage,
        data: { metrics, assessment, services: assessment.affectedServices },
      });
    }
  }
);
```

> 循环是围绕 LLM 的循环，而不是 LLM 本身。进程在步骤间重启时，已完成的步骤不会重新执行。

### Layer 2：技能（The Skill）

在这个语境下，**技能不是提示词，而是持久化工作流**：多步、可重试、可组合、可独立部署。

> Van Horn："循环是管道，真正有价值的资产是它调用的技能。" 这就是复利效应的来源 —— 系统每学会一个新技能，所有循环都变得更强大。

```typescript
export const incidentTriage = inngest.createFunction(
  { id: "incident-triage", retries: 3 },
  { event: "infra.incident.triage" },
  async ({ event, step }) => {
    const details = await step.run("fetch-detailed-metrics", async () => {
      return await fetchDetailedMetrics({ services: event.data.services });
    });

    const deploys = await step.run("fetch-deploy-history", async () => {
      return await fetchRecentDeploys({ since: hoursAgo(2) });
    });

    const analysis = await step.run("correlate-incident", async () => {
      return await callLLM({
        prompt: `Correlate these service metrics with recent deploys.
                 Identify the likely root cause and severity.
                 Metrics: ${JSON.stringify(details)}
                 Recent deploys: ${JSON.stringify(deploys)}`,
      });
    });

    await step.run("post-triage-summary", async () => {
      await slack.postMessage({
        channel: "#incidents",
        text: formatTriageSummary({
          analysis,
          affectedServices: event.data.services,
          recommendedActions: analysis.recommendations,
        }),
      });
    });

    return analysis;
  }
);
```

技能可以是中间有 LLM 的 AI 工作流，也可以是确定性代码。它是一个内置容错的工作单元。

### Layer 3：编排器（The Orchestrator）

**编排器是运行一切的引擎**：调度 cron、执行步骤、管理重试、强制并发限制、存储运行历史、在不中断运行中任务的情况下热部署新函数。

> 这是没人讨论的一层，因为它本应是隐形的。但它是基石。

传统视角：Agent = LLM + 工具。
新架构视角：**Agent = 循环 + 技能 + 编排**。LLM 和工具在循环内部，可以替换或微调，而架构保持不变。**编排层让架构成立。**

## 出错时会发生什么

### 暂时性错误：步骤级重试

技能触发时 metrics API 超时（读盘、缓存未命中），步骤重试再次命中 API，数据部分缓存后完成，技能继续执行下一步，仿佛什么都没发生。

### 不可恢复错误：失败钩子

如果 API key 过期或托管商宕机 30 分钟，所有重试耗尽。此时需要失败处理：

```typescript
export const incidentTriage = inngest.createFunction(
  {
    id: "incident-triage",
    retries: 3,
    onFailure: async ({ error, event, step }) => {
      // 重试耗尽后触发。原始事件数据仍在，什么都不丢。
      await step.run("notify-failure", async () => {
        await slack.postMessage({
          channel: "#agent-ops",
          text: `⚠️ Incident triage failed: ${error.message}. ` +
                `Will retry on next health check cycle. ` +
                `Affected services: ${event.data.services.join(", ")}`,
        });
      });
    },
  },
  { event: "infra.incident.triage" },
  async ({ event, step }) => {
    /* 与上方技能相同的逻辑 */
  }
);
```

- `onFailure` 在所有重试耗尽后触发，通知运维频道
- 事件被保留，什么都不丢
- 下次计划运行从失败处继续

> **持久化编排必须提供**：针对暂时性错误的步骤级重试 + 针对不可恢复错误的失败处理钩子。
>
> **步骤级检查点不只是正确性特性，更是省钱利器** —— 否则技能或 Agent 从头重试会重复调用 LLM、烧 token。乘以系统里 10 个或 30 个 Agent，代价昂贵。

## 会构建自己技能的 Agent

系统不是静态的，而是**设计来进化和自我扩展**的。

Agent 不只是在循环内运行 —— 它还**编写新循环并注册到编排引擎**。每个部署的函数都是一个持久化技能，独立运行，可由循环或 Agent 触发，也可按计划运行，有自己的重试逻辑。**技能会复利增长。**

这是一种"编排感知的 Agent"（orchestration-aware agent）。

### 工作机制

Agent 把编排 SDK 当作工具使用：写新函数、注册到引擎、立即开始运行。Agent 进程热加载新函数，**无需重启、不中断运行中的任务**。

### 具体示例：从需求到自治

1. **人类表达需求**（触发器）：工程师说"我们的服务经常在夜间出现延迟尖峰，没人注意直到早上。"
2. **Agent 写技能**：两个多步函数 ——
   - 健康检查循环（每 30 分钟）：拉取错误率/延迟/资源使用，LLM 将系统健康分类为 normal/degraded/critical
   - 事故分诊技能：获取详细 metrics 和最近部署历史，用 LLM 关联根因，将分诊摘要发到 Slack 并附建议行动
   - 错误处理：metrics API 挂了就退避重试；LLM 失败就回退到基于规则的严重性分类
3. **Agent 部署技能**：写入函数代码，被 sidecar 进程接管，自动注册、立即生效 —— 无需部署流水线、无需 PR
4. **技能自治运行**：每 30 分钟引擎触发健康检查；出问题就调用分诊技能；全程无人在环
5. **Agent 基于信号迭代**（关键且常被忽略的部分）：

```typescript
export const reviewSkillPerformance = inngest.createFunction(
  { id: "review-skill-performance" },
  { cron: "0 10 * * 5" }, // 每周五上午 10 点
  async ({ step }) => {
    const runs = await step.run("fetch-run-history", async () => {
      return await getInngestRuns({
        functionId: "incident-triage",
        since: daysAgo(7),
      });
    });

    const analysis = await step.run("analyze-performance", async () => {
      const successRate = runs.filter(r => r.status === "completed").length / runs.length;
      const avgDuration = average(runs.map(r => r.duration));
      const incidents = await fetchIncidentOutcomes(); // 事故是否与真实停机相关？

      return await callLLM({
        prompt: `Review this skill's performance over the past week.
                 Success rate: ${successRate}
                 Avg duration: ${avgDuration}ms
                 Incidents correlated with real outages: ${incidents.confirmed}/${incidents.total}
                 False positives: ${incidents.falsePositives}
                 Team acted on alerts: ${incidents.actedOn}/${incidents.total}

                 Should we adjust thresholds or classification? What specific changes?`,
      });
    });

    if (analysis.shouldModify) {
      await step.invoke("update-skill", {
        function: coreAgent,
        data: { prompt: `Update the incident-triage skills based on the following proposed changes: ${analysis.proposedChanges}` },
      });
    }
  }
);
```

> "review" 本身是一个函数：读取运行历史、检查事故是否与真实停机相关、把信号喂给 LLM。如果健康检查老是把某服务标记为 degraded 但团队因为阈值太敏感而忽略，review 循环就会捕捉到并更新技能。**不是魔法，是一个把 LLM 放在决策席上的 cron 任务。**

### 几个关键问题的回答

- **验证**：Agent 写的代码好坏取决于护栏。代码可类型检查；Agent 可直接调用自己注册的函数测试（与编排引擎原生交互）。虽不完美，但核心 Agent 能在它运行的系统内调试自己写的技能。review 循环兜底处理初始调试漏掉的问题。
- **反馈闭环**：进一步可让 Agent 用 `onFailure` 钩子触发自己评估失败 —— 一个持续改进的反馈环。
- **冲突**：并发控制或单例处理简单场景 ——
  ```typescript
  concurrency: [{ limit: 1, key: "event.data.service" }]
  ```
  每个服务同时只跑一个分诊。两个健康检查同时检测到同一服务问题时，编排器排队，第二次分诊等第一次完成。无重复告警、无竞态条件。

> Agent 不只是执行任务，它在为自己构建基础设施。每个技能都存续于创建它的对话之外。杀掉 Agent 进程并重启，技能继续运行；替换底层模型，技能继续运行。**Agent 是短暂的，它的产出是持久的。**

## 开发者视角

如果开发者看不到 Agent 部署了什么、无法调试出问题的部分、无法审计凌晨 3 点跑了什么 —— 整个架构就是重大负担。

编排引擎存储**每一次运行、每一个步骤、每个输入、每个输出、每次重试**。Agent 上周二部署的技能凌晨 4 点失败了？你能精确看到哪一步失败、输入是什么、抛了什么错误、重试几次后放弃。

> **这不是事后拼凑的仪表盘，而是持久化执行的内禀特性。** 每个 `step.run()` 都是一个检查点，每个检查点都可观测。当代码作者不是人类时，可观测性不是加分项 —— 它是**信任层**。

日常开发流程：早上看运行面板 → 看哪些技能夜间运行、哪些成功失败 → 如果 Agent 写的技能行为异常，可直接读代码、编辑、删除，或让 Agent 修。**Agent 写了它，但你拥有它。Agent 和它的技能仍是一座需要你打理的花园。**

## 为什么持久性是基石

Van Horn："这些必须在重启后存活。"

| 需求 | 含义 | 基础 while 循环为何失败 |
|---|---|---|
| 独立步骤重试 | 第 5 步失败时重试第 5 步，而非第 1、2 步 | 循环重启从头跑 |
| 子 Agent 生命周期 | 派生子任务、等待（可能数小时）、父取消则子取消 | 无内建父子生命周期管理 |
| 保证事件投递 | Agent 宕机期间触发的事件仍应被处理 | 进程不在运行时事件丢失 |
| 事后可观测 | 事后查看每一步、每个决策、每次重试 | 只有日志，且是短暂的 |
| 无停机热部署 | 部署新函数版本不杀运行中的任务 | 进程重启杀掉一切 |
| 并发控制 | 同时只跑 N 个技能实例 | 无内建并发原语 |

> "塞进容器里跑"能给你 uptime，给不了 correctness。崩溃后重启的容器带回了进程，但每个在途循环都从头开始，每步重执行，每次 LLM 调用重做。循环看起来在跑，但它在盲跑。

## 与现有工具的对比

- **交钥匙方案**：提供"漂亮"的一体化解决方案
- **自建**：用底层工具拼凑出自己的系统

两种选择都没错，但**正确的架构层应允许你（和你的 Agent）随时间演进**：灵活、动态、持久。

需要的是：适配 Agent 的持久化执行原语（Agent 自己能轻松编写）+ 可观测性和 API（让 Agent 本身具备编排感知）。

## 复利循环

Satya Nadella 最近的帖子命名了行业一直在感受的东西：**护城河不是模型，而是循环。**

他提出两类资本：

- **人力资本**：团队多年积累的知识与判断
- **Token 资本**：公司在基础模型之上构建的 AI 工作流、决策模式、习得技能

两者复利共生：改进的工作流产生更好的信号 → 更好的信号造就更敏锐的 AI 行为 → 更敏锐的行为释放人类注意力去做更高判断的工作。**一台爬山机。**

这正是 Agent 循环架构具体实现的东西：

1. **Agent 部署的每个持久化技能 = 被编码为可执行基础设施的组织知识**。它持久、无论有没有人看着都在运行。
2. **cron 触发的 review 循环评估技能性能并迭代** —— 这是爬山机的具体化，不是 PPT 里的飞轮图，而是一个带 cron 触发器的函数。
3. **如果技能在进程重启时死去，复利归零**。持久性让投资存续。

> Nadella 的关键点："公司应该能换掉一个'通用'模型，而不丢失学习系统里构建的'公司老手'专长。" 这就是技能库模式 —— **持久化函数不在乎是哪个 LLM 在调用它。**

## 结论

讨论已经聚焦于 Agent 做什么：循环、工具、推理、上下文工程。**下一场讨论是：什么在运行 Agent。**

三层架构：

- **循环（Loop）** —— 工作单元
- **技能（Skill）** —— 资产
- **编排引擎（Orchestrator）** —— 让前两者持久化的引擎

sidecar 模式是具体形态：Agent 编写自己的持久化技能 → 部署 → review 性能 → 迭代。不是思维实验，是一个可运行的模型。

> Inngest 被构建为这套架构的编排引擎：`step.run()`、`step.invoke()`、cron 触发器、事件驱动控制流、并发控制、全步骤级可观测性。**但架构模式比任何单一工具都大。如果你在生产环境构建 Agent 循环，定义这三层。**
>
> 基础原语今天已经存在。据此构建。
