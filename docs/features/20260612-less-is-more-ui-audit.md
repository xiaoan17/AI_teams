# Less is More：AI Teams 前端审计与优化方案

> 核心标尺（来自原文）：**少的不是数量，是优先级——用最少的元素，让最重要的信息获得最大的表达力。**
> 判断标准：如果一个元素拿掉后，表达没有变弱，它就是冗余；如果两个信号说同一件事却用不同词汇/颜色，它就是矛盾。

本应用"第一眼要让人看到什么"的答案应该是唯一的：**各个 agent 现在的状态，以及谁需要我输入**。下面所有问题都用这个标准衡量。

## 一、焦点与优先级问题（第一眼看什么）

### P1. 侧边栏顶部被低频设置占据 🔴 高优先

**现状**：侧边栏从上到下依次是 brand 块、Project 按钮、Recent 下拉、**Theme 下拉、Ambient effects 复选框**（`App.jsx:887-931`），然后才是每天高频使用的 Agents 列表。

**问题**：Theme 和光效开关是"装好后一年改一次"的设置，却永久占据侧边栏最贵的位置，把核心信息（agent 状态）往下推。Recent 下拉是一个常驻的 `<select>`，多数时候只显示"Switch recent..."占位文字——一整行只为一个低频动作服务。

**方案**：
- Theme / Ambient effects 移入一个设置入口（brand 行右侧一个 ⚙ 图标按钮，弹出小菜单），或挂到 Electron 应用菜单。
- Recent 保留为常驻下拉。这是当前产品决策：切换项目的可发现性优先于继续压缩这一行。
- 收益：侧边栏顶部从 5 行压缩到 2 行，Agents 列表上移到第一屏焦点位。

**状态**：部分完成。Theme / Ambient effects 已下沉到设置入口；Recent 按当前决策保留。

### P2. Agent 状态有四套并行的视觉信号 🔴 高优先

**现状**：同一个"agent 在干嘛"由四处表达——
1. 侧边栏 agent 行的彩色圆点（`statusClass`，`App.jsx:976`）
2. 终端卡片右上角 status pill（圆点+文字，`App.jsx:764`）
3. 卡片边框呼吸光效（绿=working、琥珀=waiting，`App.jsx:737-743`）
4. 最小化时侧边栏圆点的脉冲动画（`styles.css:389-400`）

另外活跃卡片还有 border 高亮 + focus glow（`styles.css:956-961`）。一张终端卡片上同时存在：边框色、辉光、呼吸、pill、meta 行——单卡片就有 5 个视觉变量。

**问题**：不是"信号多=更清楚"，而是观者不知道先看哪个。原文："什么都想抢注意力，观者就不知道先看哪里。" 且"working 呼吸"（有输出）和 pill 的"Ready"（状态机）是两套不同语义却共用绿色，含义互相污染。

**方案**：定义每个表面的**唯一规范信号**——
- 侧边栏：只用圆点颜色（现状保留），去掉最小化变灰等次级修饰。
- 终端卡片：status pill 作为唯一状态载体；呼吸光效只保留 **waiting_input 琥珀**这一种（它是真正需要用户行动的状态），删掉 working 绿色呼吸——"正在输出"在终端里本来就看得见，光效是重复表达。
- 活跃卡片只保留 border 变色，去掉 focus glow（或合并进光效开关后默认关闭）。

**状态**：已完成并保留一个例外。working 绿色呼吸已删除；waiting_input 琥珀呼吸保留；最小化 agent 的等待脉冲作为"隐藏面板需要用户输入"的额外信号保留。

### P3. 状态词表过多，且暴露内部诊断 🟡 中优先

**现状**：8 种状态直接展示给用户：Stopped / Starting / Ready / Needs Input / Exited / Error / Missing Runtime / Pane Missing（`App.jsx:240-249`）。

**问题**：对用户有行动意义的只有 4 类：运行中、需要输入、已停止、出错了。"Pane Missing"、"Missing Runtime" 是排障信息，"Exited" 和 "Stopped" 对用户是同一件事（都得按 ▶ 重启）。词表越大，每个词的表达力越弱。

**方案**：UI 显示收敛为 4 档（Running / Needs Input / Stopped / Error），原始状态放进 pill 的 `title` tooltip 供排障。`stoppedOrExited()` 的归类逻辑已经在这么做了，只差展示层跟上。

**状态**：已完成。

## 二、冗余元素（拿掉表达不变弱的）

### P4. 侧边栏每行三个"红绿灯"按钮 🔴 高优先

**现状**：每个运行中的 agent 行常驻 红×、黄−、绿□ 三个高饱和按钮（`App.jsx:997-1051`，`styles.css:441-488`）。

**问题**（冗余 + 矛盾叠加）：
- **颜色噪音**：N 个 agent = 3N 个高饱和色块，直接稀释了 status 圆点（红=error、黄=waiting、绿=running）的语义——同样三种颜色，左边是状态、右边是按钮，观者每次都要重新解码。
- **隐喻错位**：借用 macOS 红绿灯外观，但语义对不上——红=停掉进程（不是关窗口）、绿=聚焦（不是全屏缩放）。借来的隐喻只兑现一半，比没有隐喻更糟。
- **功能重复**：绿色按钮的"restore + select"与**点击整行**的行为完全相同（`App.jsx:1841-1849`），可直接删除。
- **死控件**：黄色按钮在已最小化时变成不可点的展示物（`window-control-muted`，title 写着 "Panel is minimized"）——一个按钮存在的唯一意义就是能按。

**方案**：
- 删掉绿□（行点击已覆盖）。
- 黄− 改为真正的 toggle（最小化⇄恢复），不再有 muted 死态。
- × 和 − 改为单色图标，**hover 该行时才显示**——这是行内操作按钮的成熟范式，平时整行只剩"圆点 + 名字"，状态色重新成为唯一的颜色信号。

**状态**：已完成。

### P5. 文档行信息过载 🟡 中优先

**现状**：每个文档行的 small 文字 = 状态徽章 + "Pinned ·" + 路径 + `toLocaleString()` 完整时间戳，右侧还常驻 +、★ 两个按钮（`App.jsx:456-485`）。

**问题**：
- "Pinned · " 文本与 ★ 图标说同一件事，纯重复。
- `2026/6/12 14:32:05` 精确到秒的时间戳没有行动价值，且把行宽撑爆走 ellipsis。
- 路径信息：树状结构本身已经表达了文件夹位置，行内再印一遍 `docs/features` 是重复（搜索过滤时例外，见方案）。
- +、★ 常驻 = 每行 2 个永远在抢注意力的控件。

**方案**：
- small 行只留：状态徽章（仅当 frontmatter 真的有 state 时，见 P6）+ 相对时间（"2h ago" / "Jun 11"）。
- 路径只在搜索/过滤态（树被压平时）显示。
- +、★ 改 hover 显示；已 pin 的 ★ 保持常显（它此时是状态不是按钮）。
- 删掉 "Pinned ·" 文本。

**状态**：已完成。

### P6. 状态徽章无中生有 + 常驻三按钮过滤器 🟡 中优先

**现状**：`documentStateLabel` 对没有 state 字段的文档**默认返回 "Todo"**（`App.jsx:332-337`），于是每一行都有彩色徽章；过滤器 All/Todo/Finish 三个按钮永久占一行（`App.jsx:1073-1085`）。

**问题**：没标 state 的文档不等于待办——这是 UI 在伪造数据。徽章一旦"每行都有"就失去表达力（全都强调=没有强调）。过滤器为一个低频动作付出一整行常驻成本。

**方案**：
- 无 state 字段 → 不渲染徽章。让 "Finish" 徽章因稀缺而醒目。
- 过滤器收进搜索框：聚焦搜索框时浮现，或做成搜索框右侧一个小下拉。

**状态**：已完成。过滤器采用搜索框右侧的小下拉。

### P7. Composer 的重复与布局 hack 🟡 中优先

**现状**：
- "Routes to: @codex" 信息行（`App.jsx:1194-1196`）与 textarea placeholder "@codex Ask an agent..." 说的是同一件事。
- Handoff `<select>` 用 `margin-top: -30px` 负边距叠到 Routes 行上（`styles.css:1043-1048`），两个信息元素挤在一条线上，靠 `pointer-events` 补丁互不干扰——典型的"用复杂度掩盖优先级没想清楚"。
- Handoff 下拉常驻展开全部文档完整路径，多数时候选的是 "No document"。

**方案**：
- 删掉 "Routes to" 常驻行；只在输入中**出现 @mention 时**于 textarea 上方浮现一行确认（此时它才携带新信息）。
- Handoff 改为 composer 顶行的 📎 图标按钮：点击弹出文档选择，选中后显示一个可×掉的 chip。armed 状态由 chip 表达，负边距 hack、`handoff-armed` 光效都可以删。
- 顺带删掉 `composer-options` 的整套负边距布局。

**状态**：已完成。

### P8. 终端头部暴露内部管线 🟢 低优先

**现状**：每张卡片 header 常显 `direct-pty %3` 这类 backend+pane 信息（`App.jsx:759-762`）。

**问题**：这是开发者排障信息，对日常使用零价值，却在每张卡片上占一行注意力。

**方案**：移入 agent 名字的 `title` tooltip；header 只留名字 + pill。

**状态**：已完成。

### P9. collapsed-brand 重复 DOM 🟢 低优先

**现状**：展开态 brand 和折叠态 collapsed-brand 是两套并存 DOM，靠 CSS 互相隐藏（`App.jsx:869-947`，`styles.css:726-759`）。

**方案**：一套 DOM + CSS 控制显隐即可；属代码层冗余，顺手清理。

**状态**：待处理，低优先。

## 三、矛盾（信号互相打架）

### P10. 术语不一致 🔴 高优先（改动成本最低、收益直接）

| 位置 | 用词 | 矛盾点 |
|---|---|---|
| 批量按钮 | "Start" / "End" | 其他所有地方都叫 Stop（`stopAgent`、"Stop agent" title、Stopped 状态） |
| 状态 pill | "Ready" | CSS 类叫 `status-running`，呼吸叫 working——同一件事三套词 |
| Files 面板 | 标题 "Files"、搜索 "Search files" | aria-label 叫 "Project documents"，代码全叫 document |
| Handoff | "Handoff" | 行话，新用户无法从词面理解（实际是"随消息附带一个任务文档"） |

**方案**：统一词表并写进文档：Start/Stop、Running、Docs（或 Files，二选一全局贯彻）；"Handoff" 改为 "Attach doc" 一类直白词。

**状态**：已完成。

### P11. placeholder 暗示与实际路由矛盾 🔴 高优先（这是行为 bug 级矛盾）

**现状**：没有活跃 agent 时 placeholder 显示 `"@all Ask the team..."`（`App.jsx:1215`），但此时 `mentionPreview` 返回 `[]`，Routes 行显示 "none"，发送时 targets 为空。

**问题**：界面同时给出两个互相矛盾的承诺——提示文案说"会发给所有人"，路由预览说"谁也不发"。用户无论相信哪个都有 50% 概率错。

**方案**：二选一并对齐：要么无活跃 agent 时默认广播（placeholder 不变，路由逻辑补上 @all 默认值）；要么 placeholder 改为 "Mention an agent to send..."。推荐后者——隐式广播副作用大。

**状态**：已完成，采用显式 mention 方案。

### P12. 红绿灯硬编码色违背主题架构承诺 🟡 中优先

**现状**：`themes.js:3-7` 注释承诺"加主题预设永远不需要碰组件逻辑"，但 `#ff5f57 / #ffbd2e / #28c840` 直接硬编码在 `styles.css:463-474`，浅色主题下这三个色块不随主题变化。

**方案**：若按 P4 把按钮单色化，此问题自动消失；否则至少收编为 token。

**状态**：已完成，随 P4 单色化消失。

## 四、字体与色彩克制（原文清单第 3、4 条）

### P13. 字号 8 档、字重 5 档 🟡 中优先

**现状**：styles.css 中字号有 10.5 / 11 / 11.5 / 12 / 12.5 / 13 / 14 / 17px 八档，字重有 500 / 650 / 700 / 750 / 800 五档。

**问题**：650 与 700、750 与 800 在 12px 下肉眼不可分——层级没有增加，只有维护噪音。原文："用字重、字号和间距建立层级"，前提是档位之间差异可感知。

**方案**：收敛为 token——
- 字号 4 档：`--fs-xs: 11px`（meta/标签）、`--fs-sm: 12.5px`（正文/行项）、`--fs-md: 14px`（标题/卡片名）、`--fs-lg: 17px`（仅 brand）。
- 字重 3 档：400（正文）、600（强调）、800（标题/标签）。
- 全局替换后视觉层级反而更清楚。

**状态**：暂缓。需要结合截图验收单独讨论，不并入当前 UI 减法批次。

### P14. 语义色彩超载 🟡 中优先

**现状**：除 accent/focus/success/warning/danger/stopped 六个语义色外，还有 doc-status / doc-finish / doc-todo / pinned-border / notice 等平行色组（`themes.js`），加上 P12 的三个硬编码色，一个界面同时存在 10+ 种有含义的颜色。

**问题**：原文："一个主色负责识别，一个辅助色负责强调，一个中性色负责承托。颜色有心理学重量，同时用太多，观者记不住核心信息。" 当前 doc-finish≈success、doc-todo≈warning、pinned≈accent，是同一语义的影子副本。

**方案**：
- `doc-finish` → 直接用 `success`；`doc-todo` → `warning`；`doc-status` → `focus`；`pinned-*` → 由 `accent` 派生（color-mix）。
- 主题文件每个预设从 30 个 token 砍到 ~20 个，三套主题同步受益。
- 最终色彩纪律：**accent（品牌/pin）+ focus（交互/选中）+ success/warning/danger（状态三色）+ 中性灰阶**，不再有第二套。

**状态**：暂缓。部分颜色语义已自然收敛，但 token 系统是否继续压缩需单独讨论。

## 五、值得保留的部分（避免矫枉过正）

原文强调"不反对复杂，反对无效复杂"。以下复杂度是**有效**的，不要砍：

- **光效系统的克制纪律**：`styles.css:952-955` 注释明确约束 max spread 18px、max alpha 0.18、周期 ≥2.8s、只作用于边框/阴影——这正是"复杂服务于单一表达"（朗香教堂式），且整体可一键关闭、尊重 `prefers-reduced-motion`。保留架构，按 P2 收窄触发条件即可。
- **主题 token 架构**：只换色不换布局的设计完全符合克制原则，问题只在 token 过多（P14）和漏网硬编码（P12）。
- **xterm 输出过滤/快照恢复**等不可见的工程复杂度：用户感知不到的复杂度不在本审计范围。
- **终端三栏自适应布局**：1/2/3 栏 + 超出横滚，规则简单、行为可预测。

## 六、实施优先级

| 批次 | 项目 | 改动量 | 一句话收益 |
|---|---|---|---|
| 第一批（高收益低风险） | P10 术语统一、P11 placeholder 矛盾、P3 状态收敛 | 小 | 消除"界面自相矛盾"，纯文案/映射层改动 |
| 第二批（焦点重塑） | P4 红绿灯改造、P2 状态信号收敛、P1 侧边栏设置下沉 | 中 | 第一眼焦点回到 agent 状态本身 |
| 第三批（密度治理） | P5/P6 文档行瘦身、P7 Composer 重构 | 中 | 高频区域的每个元素都挣得自己的位置 |
| 第四批（系统化收尾） | P13 字体 token、P14 色彩 token、P9/P12 代码清理 | 中偏大但机械 | 后续所有新功能自动继承克制的设计系统 |

**验收方法**（来自原文的眯眼测试）：改完每一批后截屏、眯眼看——第一眼看到的如果不是"哪个 agent 在等我"，就回到该批次继续减。
