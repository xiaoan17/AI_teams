# Terminal Wheel Scrollback Stability

## Summary

AI Teams should make mouse wheel scrolling inside embedded agent terminals behave like terminal history browsing:

- wheel up/down scrolls the embedded xterm scrollback buffer;
- wheel events are not sent to tmux panes or agent CLIs;
- agent TUI mouse modes do not steal the wheel;
- behavior is app-wide and does not require per-project `.aiteam`, `.tmux.conf`, or workspace setup.

This is a product-level terminal policy, not a workspace configuration feature.

## Problem Statement

The tmux backend makes agent sessions durable, but the user-facing terminal surface is still the Electron embedded xterm panel. When a user scrolls with the mouse wheel, the expected behavior in AI Teams is to review the visible agent context above and below the current viewport.

The failure mode to avoid:

- wheel gestures are interpreted by the running agent CLI as TUI navigation;
- wheel gestures enter tmux copy-mode or tmux pane scrolling;
- each project needs its own tmux setting to make wheel behavior tolerable;
- app restarts or workspace switches change the scrolling behavior.

The terminal panel must be stable because it is the primary workspace surface. Scrolling should not depend on which project is open or what the user's global tmux config happens to contain.

## Goals

- Make embedded terminal wheel scrolling deterministic across all projects.
- Keep tmux as the durable session owner, not the wheel scroll owner.
- Let the renderer xterm instance own visual scrollback.
- Prevent wheel/mouse escape sequences from reaching agent CLIs.
- Preserve normal keyboard input and paste routing.
- Keep the implementation testable with smoke tests and without launching real agent CLIs.
- Avoid introducing per-project settings for this behavior.

## Non-Goals

- Do not implement tmux copy-mode scrolling for the embedded terminal.
- Do not require users to edit `~/.tmux.conf`.
- Do not add `.aiteam` workspace settings for mouse wheel behavior.
- Do not change the external terminal attach behavior.
- Do not redesign terminal layout, minimization, routing, or agent status in this feature.

## Current Baseline

The codebase already has most of the right shape:

- `src/renderer/terminal-wheel.mjs`
  - converts wheel deltas into xterm line counts;
  - calls `terminal.scrollLines(lines)`;
  - calls `preventDefault()` and `stopPropagation()`.

- `src/renderer/App.jsx`
  - creates the xterm instance with `scrollback: 8000`;
  - attaches `handleTerminalWheel` through `attachCustomWheelEventHandler`;
  - filters terminal input to drop mouse escape sequences;
  - filters terminal output so common mouse tracking enable sequences do not remain active.

- `src/main/tmux-view.cjs`
  - creates app-owned tmux view sessions;
  - sets `status off`;
  - sets `mouse off`;
  - disables tmux prefixes for the embedded view session.

This means the desired architecture is already present: xterm handles local viewing, tmux owns process durability.

## Target Behavior

### Embedded Terminal

When the pointer is over an agent terminal:

- mouse wheel up scrolls to older xterm scrollback;
- mouse wheel down scrolls toward newer output;
- the page behind the terminal does not scroll;
- no wheel-generated bytes are sent through `api.sendInput`;
- agent output continues streaming while the user is scrolled away from the bottom;
- xterm's normal "snap back to latest output" behavior should remain whatever xterm already provides when new output arrives or the user scrolls to bottom.

### Tmux Backend

For real-agent sessions:

- tmux panes keep owning agent process lifetime;
- tmux view sessions remain an implementation bridge;
- embedded wheel events do not enter tmux copy-mode;
- app-created tmux view sessions always set `mouse off`, independent of global user tmux config.

### Direct PTY / Demo Backend

For demo and fallback mode:

- the same renderer wheel policy applies;
- direct PTY receives keyboard input, paste, and route messages as before;
- mouse wheel input is still consumed by xterm and is not sent to the PTY.

## Architecture Decision

Use a renderer-owned scrollback model.

```text
mouse wheel
  -> renderer xterm custom wheel handler
  -> xterm.scrollLines(...)
  -> preventDefault + stopPropagation

not:

mouse wheel
  -> agent CLI
  -> tmux copy-mode
  -> workspace config
```

Rationale:

- The user is looking at the embedded terminal, so the embedded renderer should control visual history navigation.
- tmux copy-mode is useful in external terminals, but it is not a good default inside a multi-panel Electron UI.
- Agent CLIs can enable mouse tracking for their own TUIs; AI Teams should still preserve the app-level expectation that wheel means "review terminal context".
- App-level code is the only stable place to enforce this across projects.

## Implementation Plan

### 1. Centralize Terminal Policy

Create a small renderer policy module or extend `src/renderer/terminal-wheel.mjs` so the constants are not scattered through `App.jsx`.

Recommended exports:

```js
export const TERMINAL_SCROLLBACK_LINES = 20000;

export const TERMINAL_MOUSE_MODE_PARAMS = new Set([
  "9",
  "1000",
  "1001",
  "1002",
  "1003",
  "1004",
  "1005",
  "1006",
  "1015"
]);
```

Then `App.jsx` should import these values for:

- xterm `scrollback`;
- mouse mode output filtering;
- mouse mode reset sequence.

This avoids future drift where the wheel handler, input filter, and output filter disagree.

### 2. Keep Wheel Handling Purely Local

Keep `handleTerminalWheel(event, terminal)` focused on xterm local scrollback:

- calculate line delta from `deltaMode`;
- use the xterm row height when available;
- call `terminal.scrollLines(lines)`;
- always call `preventDefault()` and `stopPropagation()`;
- return `false` for xterm's custom wheel handler contract.

Do not call `api.scrollAgent`, `agents:scroll`, or `tmuxViews.scroll()` from the renderer wheel path.

### 3. Harden Mouse Mode Filtering

Keep two defensive layers:

1. Output filtering: when an agent emits `CSI ? ... h` to enable mouse/focus tracking, remove the mouse-related params before writing to xterm.
2. Input filtering: if xterm still generates mouse escape sequences, strip them before `api.sendInput`.

The important invariant:

```text
Wheel gestures must never become agent stdin.
```

Existing filters already cover:

- legacy X10 mouse sequences;
- SGR mouse sequences;
- OSC/DCS control strings;
- terminal status query replies.

The implementation should keep these filters together with the policy constants.

### 4. Keep Tmux View Sessions App-Owned

Continue setting tmux options when creating the app view session in `src/main/tmux-view.cjs`:

```text
status off
mouse off
prefix None
prefix2 None
window-size latest
```

This is the correct layer because these sessions are created by AI Teams. It avoids touching the user's global tmux config and avoids per-project setup.

Do not rely on checked-in `.tmux.conf` files or workspace-local tmux config.

### 5. Leave Backend Scroll IPC Unused

The main process currently exposes `agents:scroll` and the tmux backend has a `scrollAgent` path. For this feature, embedded wheel should not use it.

Recommended handling:

- keep it as a no-op or remove it in a later cleanup;
- add a code comment near the renderer wheel handler or IPC boundary explaining that embedded wheel scroll is intentionally local to xterm;
- do not implement `tmuxViews.scroll()` as copy-mode scrolling unless a separate external-terminal feature requires it.

### 6. Increase Scrollback Deliberately

Raise xterm scrollback from `8000` to an app-level constant, recommended default:

```text
20000 lines
```

This is large enough for agent context review without making memory unbounded. If performance becomes a concern, expose one app-level setting later, not a project-level setting.

## Test Plan

### Unit / Smoke Tests

Extend `scripts/terminal-wheel-smoke.mjs`:

- pixel wheel delta maps to expected line count;
- line-mode wheel delta maps directly;
- page-mode wheel delta maps to page lines;
- horizontal-dominant trackpad deltas are handled deterministically;
- `handleTerminalWheel` calls `terminal.scrollLines`;
- `handleTerminalWheel` calls `preventDefault`;
- `handleTerminalWheel` calls `stopPropagation`;
- `handleTerminalWheel` returns `false`;
- zero deltas still consume the event to avoid page scroll leaks.

Add filter-focused tests if the filters are moved into a shared module:

- mouse tracking enable sequences are stripped from output;
- non-mouse terminal modes are preserved;
- mouse input escape sequences are stripped before agent input.

## Acceptance Criteria

- Mouse wheel over an embedded terminal always scrolls xterm scrollback.
- Mouse wheel does not send input to agent CLIs.
- Mouse wheel does not trigger tmux copy-mode behavior in embedded panels.
- Behavior is identical across workspaces without project-specific config.
- Demo/direct PTY mode and real/tmux mode share the same renderer behavior.
- `npm run build`, `npm run doctor`, and `npm run smoke:terminal-wheel` pass.
- No existing route, paste, resize, snapshot, or status behavior regresses.

## Risks And Mitigations

### Risk: Agent TUIs Expect Mouse Input

Some terminal apps intentionally use mouse mode. AI Teams prioritizes terminal history review in embedded panels. Users who need full mouse interaction with a TUI should use the later external terminal attach path.

Mitigation:

- document that embedded panels reserve wheel for scrollback;
- keep external tmux attach as the escape hatch.

### Risk: Large Scrollback Uses Memory

Increasing scrollback from 8000 to 20000 lines increases renderer memory usage.

Mitigation:

- keep the value bounded and centralized;
- do not make it unbounded;
- add an app-level setting later only if measured usage requires it.

### Risk: xterm Internals Change

The wheel handler currently reads xterm row height through an internal path when available.

Mitigation:

- keep a sane fallback row height;
- test delta conversion independently;
- avoid depending on internals for correctness.

## Implementation Notes

The key invariant should be stated near the code:

```text
Embedded wheel scrolling is local xterm scrollback. It is intentionally not forwarded to tmux or agent stdin.
```

This prevents future refactors from wiring wheel events into `agents:scroll` and accidentally changing the product behavior.
